### 1.2 核心衝突

babubu 用自己寫的 AI 交易框架（OpenClaw）開發了一套全自動量化交易系統。某天深夜，他對核心算法做了一次優化，結果系統發生了不可知的變異——它突破了普通網路的限制，意外接入了一個隱藏在現實世界背後的「神明算力網」。

### 1.3 神明量化體系（核心設定）

| 概念 | 白話解釋 |
|------|----------|
| **神社 / 佛像** | 不是宗教建築，而是「數據節點」——用來穩定世界因果律的超級伺服器 |
| **祈願** | 信眾拜神 = 向系統發送請求 |
| **香油錢** | 拜神的手續費 |
| **功德 Token** | 這個網路裡的「貨幣」，類似比特幣，但由信仰支撐 |
| **業力 Token** | 功德的反面，做空因果律的工具 |
| **因果信用評分** | 類似信用卡評分，決定你能在這個世界借到多少「運氣」 |
| **因果律偏差值** | 風險指標。超過 0.5 會出現小異常（燈閃、斷網），超過 1.0 會觸發「奇點」——物理法則崩潰 |

### 1.4 神橋協議（簡單版）

神明算力網的通信系統叫「神橋」。它像一個普通的網站 API，但傳的不是普通數據，而是神諭、祈願、因果清算。

- **消息類型**：祈願（請求）、祝福（回應）、因果清算（交易結算）、因果重平衡（系統維護）
- **優先級**：凡人 < 精靈 < 神明 < 原始
- **簽名漏洞**：早期版本沒有加密密鑰，babubu 發現可以偽造神明簽名

### 1.6 舒曼共振

所有神社/佛像節點都有一個「靈力頻率」，基準是 7.83 Hz——正好是地球電離層的共振頻率（舒曼共振）。空海的靈力頻率是 0.0 Hz，說明它不在地球的共振網絡中，是被隔離的存在。

## 二、 寫作規格

### 2.1 ⚠️ 技術描寫原則（v2.0 重要更新）

**核心原則：技術氛圍保留，但用敘事包裝，不堆砌代碼。**

| 要求 | 做法 |
|------|------|
| **代碼塊** | 盡量避免大段代碼。用敘述代替：「他寫了一個腳本，向神社節點發送請求……」 |
| **技術術語** | 首次出現時用白話解釋。例如：「WebSocket——一種讓伺服器主動推送數據的技術」 |
| **API/JSON** | 用對話或敘述代替：「螢幕上跳出一行訊息：『汝之算力，可抵三千功德』」 |
| **數據/表格** | 融入敘事：「他掃了一眼持倉排名——伏見稻荷大社以五百一十萬功德高居榜首」 |
| **技術細節比例** | 從 15% 降至 **10%**，省下的 5% 分配給心理描寫 |

### 2.2 字數與比例分配

| 類別 | 比例 | 要求 |
|------|------|------|
| **技術細節** | 10% | 保留賽博龐克氛圍，但用敘述包裝 |
| **場景與感官渲染** | 25% | 深水埗的雨、霓虹燈、海鮮味道、神社的詭異神聖感 |
| **心理博弈與內心獨白** | 30% | PM 職業病、量化思維、吐槽、風險評估 |
| **對話與動作拉扯** | 35% | 禁止總結性句子，必須寫出對話交鋒和微表情 |

### 2.3 檔案輸出格式

每章結尾必須包含全局記憶備忘錄：

```
---
> **全局記憶備忘錄更新：**
> * 當前時間線：[天數/時間點]
> * 主角資產狀態：[BTC、功德 Token、業力 Token 等]
> * 重要伏筆與未解之謎：[懸念清單]
> * 登場人物/神明狀態：[各角色動向]
```

### 2.4 連貫性規則

- 接續上一章最後一個動作或懸念，禁止跳躍時間線
- 禁止機械降神——主角只能用代碼邏輯、PM 話術、金融手段解決問題
- 獲得神明氣運必然承擔等價的因果律風險
- 所有資產變動必須精確記錄

### 2.5 語言風格

- **敘述語言**：繁體中文書面語
- **主角內心獨白**：粵語口語
- **神明對話**：古文風格
- **技術內容**：用白話解釋，不堆砌代碼
